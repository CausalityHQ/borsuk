BORSUK

Documentation

Vector search APIs, binary storage, native bindings.

Use BORSUK from Rust, Python, or TypeScript with explicit id and vector search methods, local or S3-compatible storage, and typed configuration.

API Shape

Records are ids plus vectors. You can add only vectors and let BORSUK generate ids, or add vectors with explicit ids. There are no payload references in the public API.

index.add_vectors([[1.0, 0.0], [0.0, 1.0]])
index.add_vectors_with_ids([[0.9, 0.1]], ["alpha"])

ids = index.search_ids([1.0, 0.0], k=10)
vectors = index.search_vectors([1.0, 0.0], k=10)
alpha = index.get_vector("alpha")

Typed Modes

Search options use enums for metric, search mode, and leaf mode. Leaf modes include flat-scan, sq-scan, pq-scan, graph, vamana-pq, and hybrid.

Native Packages

Python 3.12 and newer use PyO3 through maturin. Node 22, 24, and 26 use N-API. Neither package shells out through the CLI.

Binary Storage

CURRENT is a fixed binary pointer. Durable segment payloads, routing tables, manifests, segment summaries, and indexes are Parquet-backed binary data.

Decisions

BORSUK is Rust-first and exposes Python through PyO3 and TypeScript through N-API. The CLI is only an administration and debugging surface; package APIs call the Rust core directly.

The public data model is deliberately small: records are ids plus vectors. Search APIs are split by return shape, so callers choose ids with search_ids/searchIds or vectors with search_vectors/searchVectors. Report APIs expose timings, bytes, segment counts, graph work, cache hits, and resident metadata for tuning.

Persistent index state is binary and columnar. Parquet is used for manifests, segment summaries, pivots, routing, vector records, scalar/product codes, and graph blocks. CURRENT stays a fixed binary pointer so readers can atomically find the active manifest.

Architecture

The index is a set of immutable segments addressed through a small active manifest. Each segment has Parquet vector records plus compact summaries: centroid/radius routing data, id bloom filters, leaf-mode metadata, optional scalar/product code sketches, and graph blocks for graph-backed leaves.

Queries first evaluate resident manifest and routing metadata. Exact search fetches candidate segment payloads and scores vectors. Approximate search respects segment, byte, latency, and per-segment candidate budgets, then uses the selected leaf engine: flat-scan, sq-scan, pq-scan, graph, vamana-pq, or hybrid.

Local files and S3-compatible stores use the same object layout through the Rust storage layer. Optional local cache directories turn blob reads into reusable immutable object reads without changing the index format.

Functionality

The Rust crate, Python package, and TypeScript package support creating and opening indexes, adding vectors with explicit or generated ids, exact and approximate id/vector search, batch search, Float32 buffer inputs, get-by-id vector loading, stats, compaction, obsolete segment garbage collection, metric catalogs, leaf-mode catalogs, vector distance helpers, and recall_at_k/recallAtK.

The metric catalog includes Euclidean, squared Euclidean, cosine, inner product, angular, L1, L-infinity, Minkowski, histogram/distribution distances, and set-like or binary coefficient distances. Python and TypeScript expose typed enums and literal aliases for configuration rather than stringly typed free-form settings.

Storage Notes

BORSUK stores vector records in Parquet files and keeps compact routing metadata for low-RAM search. Segment summaries include a fixed-size id bloom filter and a leaf_mode field so get_vector(id), duplicate-id validation, and leaf-engine selection avoid loading segment payloads unnecessarily.

Local file paths and S3-compatible object roots use the same layout, so AWS S3, MinIO, SeaweedFS, and compatible stores exercise the same persistent format.

Testing and performance

CI runs Rust format, clippy, workspace unit and integration tests, the Rust local example, Criterion benchmark compilation, Python package tests and typing checks, TypeScript package tests, MinIO S3-compatible smoke tests, and a SeaweedFS end-to-end smoke test. Python is tested on 3.12, 3.13, and 3.14. Node is tested on 22, 24, and 26. Linux, Windows, macOS arm64, and macOS Intel are covered.

Performance coverage includes deterministic Criterion benchmarks over 10,000 64-dimensional vectors plus a performance smoke test that checks exact search and every implemented approximate leaf mode stay under the local one-second target while reporting recall, bytes read, segment pruning, graph reads, and resident metadata.
