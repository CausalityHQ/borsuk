Vector similarity search on S3-compatible object storage

BORSUK

Search hundreds of millions of vectors straight from the object storage you already pay for — with a few hundred bytes of resident memory, not a RAM-hungry cluster.

Runs on the storage you have

The whole index lives as immutable Parquet objects in S3, MinIO, SeaweedFS, GCS, or Azure. There is no always-on database tier to provision, scale, or pay for between queries — you keep your data where it already sits.

Memory stays flat as you grow

A query streams only the routing pages and vector blocks it needs. At one million vectors a search keeps roughly a few hundred bytes of index metadata resident, so a small instance serves a large index.

Answers you can trust

An exact mode returns the true nearest neighbours, and every approximate result is re-ranked on full vectors. Six leaf modes — flat-scan, sq-scan, pq-scan, graph, vamana-pq, and hybrid — trade speed against cost without giving up on recall.

How it works

BORSUK splits your vectors into many small segments and builds a layered map over them. A search walks the map to the handful of segments that could hold a neighbour, reads just those from object storage, and re-ranks the candidates on exact vectors before answering.

  • Route. A tree of centroids narrows a query to a few hundred segments instead of scanning millions of vectors.
  • Read. Only those segments — and the routing pages above them — are fetched from the blob store, so bytes read and memory stay small.
  • Re-rank. Candidates are scored on full vectors, so the returned neighbours are exact for the segments that were searched.

Why teams pick it

Serving vector search usually means standing up and paying for a dedicated in-memory fleet. BORSUK turns that into object-storage reads: pay for storage and per-query I/O, scale to large collections on small machines, and avoid a separate system to operate.

Proven at scale, honestly

A one-million-vector release gate reports 1.00 recall@10 with a few hundred bytes resident, and a completed 100-million-vector build and read probe are published with their measured numbers — every claim on this page is backed by a checked-in artifact.

Ships in three languages

One Rust engine, exposed natively to Python and TypeScript through compiled bindings — the language packages never shell out to a CLI, so behaviour and performance match across all three.

Native APIs

BORSUK is written in Rust and exposed directly through PyO3 and N-API. Start locally in whichever language you ship in:

  • Rust crate
  • Rust local example
  • Python native API
  • TypeScript native API
cargo run --locked -p borsuk --example local_index
cd python && python examples/local_index.py
cd packages/borsuk && npm run example:local
