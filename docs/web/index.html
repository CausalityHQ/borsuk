Vector search that lives in your bucket.

Why BORSUK

Serving vector search shouldn't mean renting RAM by the gigabyte.

The usual path stands up a dedicated in-memory fleet and keeps it hot whether or not anyone is querying. BORSUK turns that standing cost into plain object-storage reads: pay for storage and per-query I/O, and serve a large collection from a small machine.

In-memory ANN service

  • Whole index pinned in cluster RAM
  • Capacity sized to the dataset, not the traffic
  • A separate tier to run, patch, and page
  • Re-index and re-shard to grow

BORSUK on object storage

  • Index is immutable Parquet in the bucket
  • A query streams only the pages it needs
  • No always-on database tier between queries
  • Append segments; compaction runs in the background

Memory

RAM stays flat as the index grows.

A search walks a paged tree of centroids to the handful of segments that could hold a neighbour, then reads just those objects. At a million vectors the resident index metadata is a few hundred bytes — not gigabytes — so the same small instance serves a much larger index.

Answers

Exact when you ask, honest when you don't.

Exact mode returns the true nearest neighbours; every approximate result is re-ranked on full vectors. The production leaf mode is pq-scan — compressed, graph-free, and lowest on memory. The graph-backed modes vamana-pq and hybrid are experimental.

Writes

Incremental, not rebuild-the-world.

Inserts append immutable segments. Deletes tombstone and reclaim lazily. Oversized bubbles split, sparse ones merge, and compaction is sharded per segment so many nodes maintain the index in parallel.

Observability

Every query reports what it touched.

Bytes read, segments searched, routing pages, cache hits, and object-store requests come back on the report — so tuning is a measurement, not a guess.

How it works

Route to a few bubbles, read only those, re-rank on exact vectors.

  1. 1

    Route

    A tree of centroids narrows a query to a few hundred candidate segments instead of scanning millions of vectors.

  2. 2

    Read

    Only those segments — and the routing pages above them — are fetched from the blob store, so bytes read and memory stay small.

  3. 3

    Re-rank

    Candidates are scored on full vectors, so the neighbours returned are exact for the segments that were searched.

Where BORSUK fits

A different point in the design space.

Most vector search either pins the index in RAM, memory-maps it from local SSD, or hides it behind a managed service. BORSUK keeps it in object storage and stays an open library you run in your own process. The table compares design and deployment models — it is not a head-to-head latency benchmark, since these systems don't run on the same hardware.

System Kind Index lives in Resident RAM Updates Open / self-run
BORSUK Library Object storage (S3, GCS, Azure…) ~flat, hundreds of bytes Insert · delete · split · merge Open (BSL 1.1), in-process
SPFresh paper · SOSP '23 Research system Local NVMe SSD Working set In-place incremental (LIRE) Open prototype
DiskANN / Vamana paper Research + library Local SSD + RAM Graph kept resident Streaming (FreshDiskANN) Open, self-host
Annoy OSS Library Memory-mapped local files Mapped file pages Immutable after build Open, in-process
FAISS / hnswlib OSS Library RAM (mostly) Whole index resident Incremental add (hnswlib) Open, in-process
turbopuffer commercial Managed service Object storage + hot cache Managed for you Yes Closed SaaS
Amazon S3 Vectors commercial Managed (AWS) S3-native Managed for you Yes Closed, AWS-only
Pinecone commercial Managed service Service (RAM / SSD) Managed for you Yes Closed SaaS

The honest trade-off. An in-memory library answers a query without touching a network, and a managed service hides operations entirely — those are real advantages. BORSUK's bet is the opposite corner: keep the index in the bucket you already have, keep resident RAM flat as it grows, stay open source, and run in your own process with no separate tier to operate. It borrows the incremental-rebalancing idea from SPFresh's LIRE and graph leaves from the Vamana line of work, and lands them on immutable object storage. Reach for an in-memory engine when per-query latency is everything; reach for BORSUK when footprint, cost, and owning your own data path are.

What it costs

Mostly, you just pay for S3.

BORSUK adds no service fee and no per-query tier. Your bill is the object storage the index occupies plus the handful of GET requests a query makes. Managed vector databases bundle compute and operations — and charge a query tier on top. Quickstart

Build a local index in the language you ship in.

BORSUK is written in Rust and exposed directly through PyO3 and N-API, so the Python and TypeScript packages never shell out to a CLI — behaviour and performance match across all three. 

use borsuk::{BorsukIndex, IndexConfig, LeafMode, SearchOptions, VectorMetric, VectorRecord};

let mut index = BorsukIndex::create(IndexConfig {
    uri: "s3://my-bucket/index".into(),
    metric: VectorMetric::Cosine,
    dimensions: 768,
    segment_max_vectors: 4096,
    ram_budget_bytes: None,
})?;

index.add(vec![VectorRecord::new("doc-1", embedding)])?;
let hits = index.search_ids(&query, SearchOptions::approx(10, LeafMode::PqScan))?;
import borsuk

index = borsuk.create(uri="s3://my-bucket/index", metric="cosine", dim=768)
index.add([embedding], ids=["doc-1"])

hits = index.search_ids(query, k=10, mode="approx", leaf_mode="pq-scan")
import { create } from "borsuk";

const index = await create({ uri: "s3://my-bucket/index", metric: "cosine", dimensions: 768 });
await index.add([embedding], { ids: ["doc-1"] });

const hits = await index.searchIds(query, { k: 10, mode: "approx", leafMode: "pq-scan" });

Or run the bundled example end to end: cargo run --locked -p borsuk --example local_index
