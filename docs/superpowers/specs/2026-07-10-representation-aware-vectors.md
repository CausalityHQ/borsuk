# BORSUK — Representation-aware vector math (remove densify-on-read)

**Why:** Phase C stored a vector dense-or-sparse but DENSIFIED sparse on read, so a high-dimensional lexical vector (huge vocab, few non-zeros) can't be handled — you'd allocate a D-element array. User (2026-07-10): "why do we densify? we should not — smart algos to handle dense vs sparse, sparse vs dense, etc." Correct. Make all vector ops representation-aware; never densify for the dot-family metrics.

**Math (no densify):**
- `dot(dense,dense)=Σa[i]b[i]`; `dot(sparse,dense)=Σ_{(i,v)∈sparse} v·dense[i]` (O(nnz)); `dot(sparse,sparse)`=merge-join over sorted indices (O(nnz_a+nnz_b)).
- Store per-vector squared norm `‖v‖²`. Dot-family metrics via dot+norms: inner-product=`-dot`; cosine=`1-dot/(‖a‖‖b‖)`; euclidean=`√(‖a‖²+‖b‖²-2dot)`; squared-euclidean; angular=acos(cos)/π. So a 100k-dim, 20-nnz vector is only ever touched 20 elems at a time.
- NON-dot metrics (manhattan, chebyshev, minkowski, set/binary, distribution, DTW) do NOT reduce to dot → stay DENSE-ONLY (never used with high-dim lexical). Sparse-stored vectors are only valid with dot-family metrics; validate at create.

**Stages (each gated + pushed green; rigorous reference cross-checks):**
- **RA1a** (safe, isolated): representation-aware `dot`/`squared_norm` primitives for all 4 dense/sparse combos + unit tests vs a reference. No engine wiring. (Like sparse.rs Phase 1a.)
- **RA1b**: keep the stored representation in memory (VectorRecord holds a dense-or-sparse `Vector`, NOT densified on read); route dot-family metric computation + LEAF exact-rerank through the representation-aware primitives (no densify at query time). Non-dot metrics densify on demand. Recall gates must stay 1.0. Correctness test: a high-dim sparse index (D=100000, nnz~20) builds + searches with NO large allocation and results match a brute-force sparse-dot reference.
- **RA2**: centroids stored in the compact representation (sparse when the segment's term-union is small); representation-aware triangle-inequality routing lower bound; BYPASS PQ for sparse segments (exact sparse-dot leaf is O(nnz)). Routing must prune correctly for sparse (recall preserved).
- **RA3**: lift the lexical-sparse caveat in docs; make the Pinecone/Qdrant SPARSE drop-in real (sparse_values → a named sparse vector, no densify). Update adapters + docs + landscape.

**INVERTED INDEX (user 2026-07-10: "for what do we need the inverted index? if needed add it also"):** For HIGH-DIM sparse (lexical/SPLADE) the metric-tree centroid/radius routing degrades (curse of dimensionality) — the right retrieval is an INVERTED INDEX: term → posting list of (record, weight). A sparse query gathers only records sharing its query terms (like BM25), scores them with representation-aware `sparse_dot` (RA1a, no densify), top-k. So:
- **INV** stage: a named vector declared SPARSE is backed by an inverted-index retrieval path (bring back the removed `sidx` machinery — SparseIndex term postings + on-demand sidecar), NOT the metric tree. `search(vector=sparse_name, query)` = inverted-index candidate gather + RA1a sparse-dot scoring + top-k. This is the efficient, correct high-dim sparse path and makes the Pinecone/Qdrant sparse drop-in real. Dense named vectors keep the metric tree. The index/named-vector config picks the backend (tree for dense, inverted for sparse) — or a heuristic.

**SIMPLER ARCHITECTURE (chosen 2026-07-10):** A SPARSE named vector uses the INVERTED INDEX as its backend directly — NOT the metric tree — so the risky metric-tree representation-aware refactor (RA1b/RA2) is UNNEEDED for sparse. Dense named vectors + the primary keep the metric tree (densify-on-read is fine for dense/moderate). A named vector declared `kind: Sparse` (extend VectorSpec with a kind/backend) gets an inverted-index sub-index; add stores its sparse vectors + term postings; `search(vector=sparse_name, query)` gathers candidates sharing query terms via the postings, scores with RA1a `sparse_dot` (no densify), top-k. RA1a primitives are all the math needed. This is the removed `sidx`/`SparseIndex` machinery brought back as a first-class sparse-named-vector backend.
